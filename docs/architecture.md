# 架构设计与 2026 痛点治理

## 1. 范围说明

本项目定位为自研协议原型（NebulaStore），目标是构建纯自研的去中心化存储系统。
控制面能力与协议机制共同构成完整方案，不依赖第三方存储网络编排。

## 2. 控制面组件

- 域名注册中心：维护 `domain -> owner -> deployment -> expiry` 映射。
- 策略集合：统一表达检索、耐久、经济、合规四类治理参数。
- 状态快照：本地可重复、可审计、可回放的 JSON 状态。
- 编排入口：CLI 与 HTTP API 统一调用同一套核心逻辑。
- 协议引擎：内置检索、成本、耐久、调度、合规五类核心机制。
- 规划引擎：输出分片放置、节点信誉排序、挑战窗口策略。
- 修复引擎：挑战失败自动入队并执行重复制状态机。
- 经济惩罚引擎：按作恶/离线类型执行罚没和恢复期。
- 分层迁移引擎：根据热度在 hot-edge 与 cold-archive 间迁移。
- 工作流编排引擎：串联 plan/repair/penalty/migrate 并输出统一摘要。
- 多目标优化引擎：输出 Pareto 前沿，用于成本-时延-可靠性权衡。

## 3. 与行业核心问题的映射

### 检索速度 / 延迟慢

治理策略：

- 在 `retrieval_profile` 中定义 `target_p95_ms` 目标。
- 明确加速层：`edge-cache`、`gateway-cache`、`origin-fallback`。
- 支持混合架构：热路径缓存或中心化加速，冷路径保留去中心化。

### 成本波动大 / 经济模型不稳定

治理策略：

- 通过 `quote_valid_hours` 锁定短时成本窗口。
- 波动超阈值才触发再平衡，降低频繁迁移成本。
- 采用 `reserved + spot` 混合模式平滑价格波动。

### 可用性 / 永久性不可靠

治理策略：

- 设定 `min_replicas=5` 的跨节点池副本基线。
- 使用纠删码与修复阈值，提前发现并修复风险。
- 可扩展为定时重复制任务，避免长期掉线累积。

### 节点异构导致性能不均

治理策略：

- 节点健康评分调度，按在线率、时延、吞吐动态分配负载。
- 读取路径基于健康度做故障切换与热点迁移。

### 存储证明复杂与持续性验证成本高

治理策略：

- 按 24 小时窗口执行抽样挑战，降低链上验证压力。
- 副本数越高，挑战抽样比例越高，保证高价值数据更安全。
- 使用修复触发阈值，将挑战结果直接联动到重复制流程。

### 节点作恶治理与经济激励不足

治理策略：

- 对作恶证明执行较高比例罚没并拉长恢复周期。
- 对长离线执行较低比例罚没并给出快速恢复通道。
- 将信誉恢复与连续稳定在线行为绑定，防止短期刷分。

### 冷热数据混放导致性能与成本失衡

治理策略：

- 用热度占比驱动自动迁移执行器。
- 热分片迁入 hot-edge 提升时延表现，冷分片迁入 cold-archive 降成本。
- 迁移执行结果联动延迟与成本指标，形成闭环调优。
- 避免把关键负载集中到低质量节点。

### 用户体验与采用门槛高

治理策略：

- 统一动词语义的 CLI + HTTP API。
- 快照机制让本地调试和回归测试更简单。
- 续期/删除必须带所有者，减少误操作风险。

### 合规、隐私与监管摩擦

治理策略：

- 以客户端加密作为默认基线。
- 通过区域 pinning 限定数据主权边界。
- 开启审计轨迹字段，便于监管与审计留痕。

## 4. 2026 年可落地的混合策略

- 高频热数据：CDN + 中心化对象存储。
- 冷归档、存证、抗审查内容：去中心化存储网络。
- 域名与策略在控制面统一治理，协调冷热两类路径。

## 5. 后续工程化方向

- 把当前 NebulaStore 原型扩展为真实 P2P 网络与存储证明协议。
- 增加健康探测、时延遥测、成功率监控。
- 增加成本采集器与自动再平衡执行器。
- 增加密钥管理、访问控制与策略审批流程。
